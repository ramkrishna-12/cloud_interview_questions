Q1
Explain the full AWS VPC architecture you would design for a production multi-tier application. Include subnets, routing, NAT, and security layers.
HardVPCNetworking
Optimum Answer
VPC design (production-grade, 2 AZs minimum):

Subnet layout per AZ:
  • Public subnet (e.g., 10.0.1.0/24): ALB, NAT Gateway, Bastion host only. Route table: 0.0.0.0/0 → Internet Gateway.
  • Private App subnet (e.g., 10.0.10.0/24): EC2/EKS nodes, app servers. Route table: 0.0.0.0/0 → NAT Gateway (outbound only).
  • Private DB subnet (e.g., 10.0.20.0/24): RDS, Aurora, ElastiCache. No internet route at all. Only app subnet SG can reach port 5432/3306.
Key components:
  • Internet Gateway (IGW): Attached to VPC, used by public subnets only
  • NAT Gateway: One per AZ (for HA) in the public subnet — allows private instances to pull updates/packages without being publicly reachable
  • Security Groups: Stateful, instance-level — ALB SG allows 80/443 from 0.0.0.0/0; App SG allows 8000 from ALB SG only; DB SG allows 5432 from App SG only
  • NACLs: Stateless subnet-level — use to block known malicious CIDRs
  • VPC Flow Logs: Enable on all subnets, send to CloudWatch Logs / S3 for audit
  • VPC Endpoints: S3 and DynamoDB gateway endpoints — keep S3 traffic inside AWS backbone, not through NAT (cost + security)
Your real experience: Reference your 3-tier compliance app VPC design and blue-green migration with this exact layout. Mention specific CIDR blocks you used.
Q2
What is AWS EKS? How does it differ from ECS and self-managed Kubernetes? How do you provision an EKS cluster with managed node groups?
HardEKSKubernetes
Optimum Answer
EKS (Elastic Kubernetes Service) — AWS-managed Kubernetes control plane. You pay for the control plane ($0.10/hr); AWS handles etcd, API server HA, and control plane upgrades. You manage worker nodes (EC2 Managed Node Groups or Fargate).

EKS vs ECS vs self-managed K8s:
  • ECS: AWS-proprietary container orchestrator. Simpler, no YAML manifests, tightly integrated with AWS. No portability. Choose ECS for AWS-only, simpler workloads.
  • EKS: Standard Kubernetes — portable, ecosystem (Helm, Argo CD, Prometheus). More complex but industry-standard. Choose for microservices, multi-cloud roadmap, existing K8s expertise.
  • Self-managed K8s: You run kubeadm on EC2. Full control, no SLA from AWS on control plane. Only for very specific compliance needs.
Provisioning EKS with Terraform: 
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name    = "prod-cluster"
  cluster_version = "1.29"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnets

  eks_managed_node_groups = {
    app = {
      instance_types = ["t3.medium"]
      min_size       = 2
      max_size       = 6
      desired_size   = 2
    }
  }
  # Enable IRSA for pod-level IAM roles
  enable_irsa = true
}
After provisioning: aws eks update-kubeconfig --name prod-cluster --region ap-south-1 then kubectl get nodes to verify.
Q3
Explain RDS Aurora vs standard RDS PostgreSQL. When would you choose Aurora over RDS?
MediumRDSAuroraDatabase
Optimum Answer
Standard RDS PostgreSQL: Managed single EC2 instance running PostgreSQL. Multi-AZ = synchronous standby replica (failover ~60s). Read replicas = asynchronous. Storage provisioned and scaled manually (up to 64TB). Familiar tuning, straightforward pricing.

Aurora PostgreSQL: Cloud-native re-architecture. Storage layer is distributed across 6 copies in 3 AZs automatically — separate from compute. Key differences:
  • Storage auto-scales from 10GB to 128TB in 10GB increments — no provisioning
  • Faster failover: ~30 seconds (vs 60–120s for RDS Multi-AZ) because the replica already has the same distributed storage
  • Up to 15 read replicas with near-zero replication lag (all share same storage layer)
  • Aurora Serverless v2: Auto-scales compute capacity from 0.5 to 128 ACUs in seconds — perfect for variable workloads
  • Higher cost: ~20% more than equivalent RDS, but often worth it for HA requirements
Choose Aurora when: you need sub-30s failover, auto-storage scaling, or >5 read replicas. Choose standard RDS when: budget is tight, simpler workloads, or specific PostgreSQL version/extension compatibility matters.
Q4
What is AWS Lambda? Describe how you would use Lambda in an event-driven architecture for processing S3 file uploads.
MediumLambdaServerless
Optimum Answer
AWS Lambda is a serverless compute service — you upload code, Lambda runs it in response to events, scales to zero when idle, bills per 100ms of execution. No servers to patch or manage.

S3 → Lambda event-driven pipeline:
  • User/system uploads a file to s3://raw-uploads/
  • S3 Event Notification triggers the Lambda function on s3:ObjectCreated:*
  • Lambda receives the event, reads the S3 object, processes it (validate, transform, enrich)
  • Lambda writes output to s3://processed/ AND inserts a record into RDS/DynamoDB
  • On failure: Lambda sends message to SQS Dead Letter Queue (DLQ) for retry
import boto3, json

def handler(event, context):
    s3 = boto3.client('s3')
    for record in event['Records']:
        bucket = record['s3']['bucket']['name']
        key    = record['s3']['object']['key']
        obj    = s3.get_object(Bucket=bucket, Key=key)
        data   = obj['Body'].read()
        process_and_store(data)
    return {'statusCode': 200}
Key Lambda settings for production: Memory (128MB–10GB, directly affects CPU too), Timeout (max 15 min), Concurrency limits (reserve concurrency to prevent DB connection exhaustion), and Layers for shared dependencies.
Q5
What is AWS IoT Core? Describe its architecture: devices, MQTT, Rules Engine, and integration with other AWS services.
MediumAWS IoTMQTT
Optimum Answer
AWS IoT Core is a managed cloud service that lets connected devices (sensors, machines, edge devices) communicate with AWS services securely at scale without managing broker infrastructure.

Architecture components:
  • Device SDK: Runs on device (Python, C, Node.js). Handles TLS mutual auth and MQTT protocol
  • Message Broker: MQTT broker in the cloud. Devices publish to topics (e.g., factory/machine-01/temp). Other devices or backend can subscribe
  • Device Registry: Catalog of all registered things with metadata and attributes
  • Device Shadow: JSON document representing last known state of a device — even when device is offline, you can read/update the shadow
  • Rules Engine: SQL-like rules that trigger actions on messages matching a topic. E.g., "if temperature > 80°C → invoke Lambda + write to DynamoDB + send SNS alert"
Integration flow: Device → MQTT over TLS → IoT Core Broker → Rules Engine → Lambda / DynamoDB / Kinesis / S3 / CloudWatch

Security model: X.509 certificates per device (mutual TLS), IoT Policies (like IAM policies) scoped per device.
Q6
How do you approach cloud cost optimization on AWS? Give specific techniques for compute, storage, and data transfer.
MediumCost OptimizationFinOps
Optimum Answer
Compute:
  • Reserved Instances / Savings Plans: 1–3 year commitment saves 40–72% vs On-Demand. Use for steady baseline load
  • Spot Instances: 70–90% discount for fault-tolerant workloads (batch jobs, Celery workers, CI/CD runners)
  • Right-sizing: Use AWS Compute Optimizer recommendations. Most orgs are over-provisioned by 2–3x. Check CloudWatch CPU/memory over 14 days
  • Auto Scaling + scheduled scaling: Scale down to minimum outside business hours
Storage:
  • S3 Intelligent-Tiering: Auto-moves objects to cheaper tiers if not accessed for 30 days
  • EBS volume cleanup: Delete unattached EBS volumes and unused snapshots (common cost leak)
  • S3 Lifecycle policies: Transition to Glacier Deep Archive after 90 days for archival data
Data Transfer:
  • VPC Endpoints: S3/DynamoDB gateway endpoints eliminate NAT Gateway data processing charges for S3 traffic
  • CloudFront: Reduces origin data transfer cost by caching at edge
  • Keep traffic same-AZ: Cross-AZ data transfer costs $0.01/GB each way — place compute near data
Tools: AWS Cost Explorer, AWS Budgets with alerts, Trusted Advisor, Infracost (in Terraform CI/CD to flag cost increases before apply).
Q7
Describe a disaster recovery strategy on AWS. Explain RTO/RPO and the difference between Pilot Light, Warm Standby, and Multi-Site Active-Active.
HardDisaster RecoveryHA
Optimum Answer
RTO (Recovery Time Objective): How long you can be down. RPO (Recovery Point Objective): How much data loss is acceptable.

  • Backup & Restore: Cheapest. Periodic snapshots to S3 Glacier. RTO: hours. RPO: hours. Use for non-critical systems.
  • Pilot Light: Core services (DB) always running in DR region, but minimal compute. On disaster: scale up compute from AMIs, point DNS to DR. RTO: 30–60 min. RPO: minutes (depends on replication lag).
  • Warm Standby: Scaled-down but fully functional environment always running in DR. On disaster: scale up to production size. RTO: minutes. RPO: seconds.
  • Multi-Site Active-Active: Full production capacity in 2+ regions simultaneously. Traffic split via Route 53 latency/geolocation routing. On failure: Route 53 health checks redirect 100% traffic to healthy region. RTO: near-zero. RPO: near-zero. Most expensive.
For most production workloads: Warm Standby with Aurora Global Database (cross-region replication with ~1s lag) is the best cost/RTO balance. RDS automated backups (35-day retention) + cross-region snapshot copy for RPO protection.

Q8
Compare GKE (Google Kubernetes Engine) with AWS EKS. What unique features does GKE offer?
MediumGKEKubernetes
Optimum Answer
Both are managed Kubernetes control planes. Key comparison:
  • GKE Autopilot: Google manages not just the control plane but ALSO the nodes — you just deploy pods, pay per pod resource usage. No node group management. EKS has no equivalent (Fargate is close but less seamless).
  • Cluster upgrades: GKE has Release Channels (Rapid/Regular/Stable) — auto-upgrades with controlled rollout. EKS upgrades are more manual (you trigger control plane upgrade, then separately upgrade node groups).
  • Networking: GKE uses VPC-native networking by default — pod IPs are real VPC IPs (no overlay needed). EKS requires the AWS VPC CNI plugin, and pod IP density per node is limited by ENI limits.
  • Workload Identity: GKE's equivalent of IRSA — maps K8s ServiceAccounts to GCP Service Accounts for pod-level IAM without static credentials.
  • Cost: GKE charges $0.10/hr for the control plane (same as EKS). Autopilot mode can be cheaper for variable workloads.
GKE is generally considered more developer-friendly and has tighter Kubernetes integration. EKS has better AWS ecosystem integration (IAM, ALB controller, etc.).
Q9
How do you deploy a containerised application to Cloud Run and connect it to Cloud SQL? What are Cloud Run's concurrency and scaling model?
HardCloud RunCloud SQL
Optimum Answer
Deployment: 
# Build and push to Artifact Registry
gcloud builds submit --tag gcr.io/PROJECT/myapp

# Deploy to Cloud Run with Cloud SQL connection
gcloud run deploy myapp \
  --image gcr.io/PROJECT/myapp \
  --platform managed \
  --region asia-south1 \
  --add-cloudsql-instances PROJECT:asia-south1:mydb \
  --set-env-vars DB_HOST=/cloudsql/PROJECT:asia-south1:mydb \
  --set-secrets DB_PASS=db-password:latest \
  --service-account myapp-sa@PROJECT.iam.gserviceaccount.com \
  --min-instances 1 \
  --max-instances 20 \
  --concurrency 80
Cloud Run Scaling Model:
  • Concurrency: Each container instance handles up to N requests simultaneously (default 80, max 1000). Unlike Lambda (1 request per container), Cloud Run can handle multiple concurrent requests per instance.
  • Scaling: Cloud Run scales instances based on incoming requests. With --min-instances 0 it scales to zero (cold starts). With --min-instances 1 always one warm instance.
  • CPU allocation: Default is CPU only allocated during request processing. Use --cpu-always-on for background work (Celery-like patterns).
Cloud SQL Auth Proxy runs as a sidecar in Cloud Run automatically when you use --add-cloudsql-instances — handles TLS connection without managing the proxy yourself.
Q10
What is Cloud Storage and how do GCP Storage Classes work? How does it compare to AWS S3?
EasyCloud StorageGCP
Optimum Answer
GCP Cloud Storage is GCP's object storage — analogous to S3. Objects stored in buckets. Globally unique bucket names. Strong consistency (unlike S3's eventual consistency era — both are now strongly consistent).

Storage Classes:
  • Standard: Hot data, frequent access. Highest storage cost, no retrieval cost.
  • Nearline: Access <1/month. 30-day minimum. Retrieval cost applies.
  • Coldline: Access <1/quarter. 90-day minimum. Higher retrieval cost.
  • Archive: <1/year access. 365-day minimum. Lowest storage, highest retrieval cost.
Use Object Lifecycle Management to auto-transition objects between classes (just like S3 Lifecycle Policies).

vs AWS S3: Functionally equivalent. GCP has no S3 Glacier equivalent (Archive class fills that role). GCP has stronger IAM integration. S3 has more ecosystem tooling (AWS Data Sync, Transfer Acceleration). Both support versioning, signed URLs, and event notifications (GCP → Pub/Sub; S3 → SNS/Lambda).

Q13
Explain the core Kubernetes objects: Pod, Deployment, Service, Ingress, ConfigMap, and Secret. How do they relate?
MediumK8s Objects
Optimum Answer
  • Pod: Smallest deployable unit. One or more containers sharing network namespace and volumes. Never create Pods directly in production — use Deployments.
  • Deployment: Manages a ReplicaSet (desired number of Pod replicas). Handles rolling updates and rollbacks. kubectl rollout undo deployment/myapp to revert.
  • Service: Stable network endpoint for a group of Pods (selected by label). Types: ClusterIP (internal), NodePort, LoadBalancer (creates cloud LB), ExternalName. Pods come and go; Service IP stays constant.
  • Ingress: HTTP/HTTPS routing rules at layer 7. Maps paths/hostnames to Services. Requires an Ingress Controller (nginx-ingress, AWS ALB Ingress Controller). E.g., api.company.com → service:api:8000
  • ConfigMap: Non-sensitive config data (env vars, config files) injected into Pods. Decouples config from container image.
  • Secret: Base64-encoded (NOT encrypted by default — use Sealed Secrets or External Secrets Operator in production) sensitive data. Same injection mechanisms as ConfigMap.
Relationship: Ingress → Service → Deployment → ReplicaSet → Pods. ConfigMap/Secret injected into Pods as env vars or volume mounts.
Kubernetes Secrets are only base64 encoded by default, NOT encrypted at rest. Use AWS Secrets Manager + External Secrets Operator or Sealed Secrets for production.
Q14
What is Argo CD and how does it implement GitOps? Explain the Application CRD, sync policies, and health checks.
HardArgo CDGitOps
Optimum Answer
GitOps principle: Git is the single source of truth for desired cluster state. Changes are made via PRs to Git; an operator (Argo CD) continuously reconciles actual cluster state to match Git.

Argo CD Application CRD: 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/infra-repo
    targetRevision: main
    path: k8s/overlays/prod   # Kustomize or Helm path
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true      # Delete resources removed from Git
      selfHeal: true   # Revert manual kubectl changes
    syncOptions:
      - CreateNamespace=true
Sync policies:
  • Manual sync: Argo CD shows diff, you click sync manually (safer for prod)
  • Automated sync: Argo CD polls git (every 3 min) or gets webhook trigger and auto-applies changes
  • Sync Waves: Annotate resources with argocd.argoproj.io/sync-wave: "1" to control order (e.g., deploy DB migrations before app)
Health checks: Argo CD has built-in health assessments for Deployments (checks rollout status), Services, Ingresses, PVCs etc. Custom health checks via Lua scripts for CRDs.

What is Helm and how does it relate to Kubernetes? Explain Charts, Values, and Release management.
MediumHelmPackage Manager
Optimum Answer
Helm is the package manager for Kubernetes — it templates and manages groups of K8s manifests as versioned, reusable packages called Charts.

Chart structure: 
mychart/
  Chart.yaml        # Metadata: name, version, description
  values.yaml       # Default configuration values
  templates/        # Go-template K8s manifests
    deployment.yaml
    service.yaml
    ingress.yaml
Key commands: 
# Install a chart (creates a Release)
helm install myapp ./mychart -f values.prod.yaml -n production

# Upgrade with new image tag
helm upgrade myapp ./mychart --set image.tag=v1.2.4

# Rollback to previous release
helm rollback myapp 1

# View all releases
helm list -A
Values override hierarchy: values.yaml (defaults) → -f values.prod.yaml (file override) → --set key=val (CLI override, highest priority).

Helm is used heavily with Argo CD — Argo CD can deploy Helm charts directly by pointing at a chart repo and values file in Git.
Q17
How does Kubernetes networking work? Explain Pod-to-Pod, Pod-to-Service, and external traffic routing.
HardK8s NetworkingCNI
Optimum Answer
K8s networking model: Every Pod gets its own IP. All Pods can communicate with all other Pods without NAT (flat network). The CNI plugin (Calico, Cilium, AWS VPC CNI, Flannel) implements this.

Pod-to-Pod: Direct IP communication using the CNI network. On EKS with AWS VPC CNI, pod IPs are real VPC IPs — routing happens via VPC routing tables.

Pod-to-Service: Service gets a stable ClusterIP (virtual IP). kube-proxy on each node maintains iptables/IPVS rules that DNAT the ClusterIP to one of the backend Pod IPs (load balanced). DNS: my-service.my-namespace.svc.cluster.local resolves to the ClusterIP via CoreDNS.

External traffic:
  • NodePort: Opens a port on every node (30000–32767). External traffic hits any node IP:port → iptables routes to Pod.
  • LoadBalancer: Provisions a cloud load balancer (NLB/ALB) that routes to NodePort.
  • Ingress: Single L7 entry point. Ingress Controller (nginx) runs as a Pod, watches Ingress objects, routes HTTP traffic to Services by hostname/path.
Network Policies: K8s firewall rules at Pod level — restrict which Pods can talk to which. Default: allow all. Apply least-privilege NetworkPolicy for production.

Q20
How do you troubleshoot a Kubernetes Pod stuck in CrashLoopBackOff? Give your full diagnostic workflow.
MediumK8s Troubleshooting
Optimum Answer
Systematic kubectl diagnostic workflow: 
# 1. Describe the pod — look at Events section
kubectl describe pod <pod-name> -n production

# 2. Get logs from current container
kubectl logs <pod-name> -n production

# 3. Get logs from previous crashed container
kubectl logs <pod-name> -n production --previous

# 4. Check resource pressure on the node
kubectl top nodes
kubectl top pods -n production

# 5. Exec into a running (init) container for debugging
kubectl exec -it <pod-name> -n production -- /bin/sh

# 6. Check events cluster-wide
kubectl get events -n production --sort-by='.lastTimestamp'
Common causes and fixes:
  • OOMKilled (exit 137): Memory limit too low → increase resources.limits.memory
  • Missing env var / secret: describe pod shows "Error: secret not found" → create the missing Secret
  • Liveness probe failing: App takes too long to start → increase initialDelaySeconds
  • Image pull error: Wrong image tag or registry credentials → check imagePullSecrets
  • Config errors: App crashes at startup due to bad config → check --previous logs for the actual error

Compare Jenkins, GitLab CI, and GitHub Actions. When would you choose each for an enterprise environment?

Q24
Explain deployment strategies: Rolling Update, Blue/Green, Canary in the context of Kubernetes. How does Argo Rollouts help?
HardDeployment StrategiesArgo Rollouts
Optimum Answer
Rolling Update (K8s default): Gradually replaces old Pods with new ones. Controlled by maxSurge and maxUnavailable. Zero downtime but both versions run simultaneously. Rollback: kubectl rollout undo.

Blue/Green: Two full Deployments (blue=old, green=new). Switch Service selector from blue to green in one atomic operation. Instant rollback (switch selector back). Requires 2x resources during transition.

Canary: Route small % of traffic to new version. Monitor metrics. Gradually increase. Native K8s canary needs two Deployments + Ingress weight annotations. Complex to manage manually.

Argo Rollouts is a K8s controller that adds advanced deployment strategies as a CRD: 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 10        # 10% traffic to new version
      - pause: { duration: 1h }
      - setWeight: 50
      - analysis:            # Run metric analysis
          templates:
          - templateName: success-rate
      - setWeight: 100
Integrates with Prometheus metrics to automatically roll back if error rate spikes during canary analysis.

MediumActive DirectoryGPO
Optimum Answer
GPO (Group Policy Object): A collection of settings applied to computers and users in an AD environment. Linked to Sites, Domains, or OUs. Applied in LSDOU order (Local → Site → Domain → OU). Later-applied policies win by default.

Common GPO uses:
  • Security baselines (password policy, account lockout, Windows Firewall rules)
  • Software deployment (MSI packages pushed to computers)
  • Drive/printer mapping, desktop wallpaper, browser settings
  • Restrict Control Panel, USB access, removable media
GPO troubleshooting commands: 
# Force immediate GPO refresh
gpupdate /force

# See which GPOs are applied to current machine/user
gpresult /R

# Detailed HTML report of GPO application
gpresult /H C:\gpreport.html /F

# Check GPO event logs
Get-WinEvent -LogName 'Microsoft-Windows-GroupPolicy/Operational' |
  Where-Object {$_.LevelDisplayName -eq 'Error'} | Select -First 10
Common issues: GPO linked to wrong OU, WMI filter blocking application, security filtering excluding the computer/user account, replication latency between DCs.
Q29
Compare Nginx and Apache as web servers. When would you choose each, and how do you configure Nginx as a reverse proxy with SSL termination?
HardNginxApacheSSL
Optimum Answer
Apache: Process/thread-based model. Excellent for dynamic content with mod_php (PHP embedded in process). .htaccess per-directory config (flexible but slow — reads file on every request). Rich module ecosystem. Better for shared hosting or legacy PHP apps.

Nginx: Event-driven, async non-blocking. Handles thousands of concurrent connections with very low memory. No per-directory config files. Excellent as a reverse proxy, load balancer, and static file server. Better for high-concurrency and modern architectures.

Nginx reverse proxy + SSL termination config: 
server {
    listen 443 ssl http2;
    server_name api.company.com;

    ssl_certificate     /etc/ssl/certs/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    # Security headers
    add_header Strict-Transport-Security "max-age=31536000" always;
    add_header X-Frame-Options DENY;
    add_header X-Content-Type-Options nosniff;

        proxy_pass         http://127.0.0.1:8000;

        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;

Key postgresql.conf tunings: 
max_connections = 200           # Use PgBouncer if >200
shared_buffers = 4GB           # 25% of RAM
effective_cache_size = 12GB    # 75% of RAM
work_mem = 64MB                # Per sort/hash operation
wal_buffers = 64MB
checkpoint_completion_target = 0.9
log_min_duration_statement = 1000  # Log queries >1s
log_checkpoints = on
autovacuum = on                # Never disable this
Connection pooling with PgBouncer: Place PgBouncer between app and PostgreSQL. Transaction mode (most efficient) — connection returned to pool after each transaction. App connects to PgBouncer on port 6432; PgBouncer maintains a pool of 20–30 real PG connections shared by 200+ app connections.

VACUUM: PostgreSQL uses MVCC — dead tuples accumulate after updates/deletes. autovacuum reclaims space. Monitor with: SELECT relname, n_dead_tup, last_autovacuum FROM pg_stat_user_tables ORDER BY n_dead_tup DESC;

HardSQL ServerAlways On
Optimum Answer
SQL Server Always On Availability Groups (AG) — SQL Server's HA/DR solution. A group of databases that fail over together as a unit across multiple SQL Server instances.

Components:
  • Primary replica: Read-write. Sends log records to secondary replicas.
  • Secondary replicas: Synchronous (data loss = 0, failover ~10s) or Asynchronous (possible data loss, used for DR across sites) modes.
  • Listener: Virtual IP/DNS name that clients connect to — automatically routes to current primary after failover
  • Windows Server Failover Cluster (WSFC): Required underpinning for automatic failover
Backup strategy: 
-- Full backup weekly (Sunday 2 AM)
BACKUP DATABASE [ProdDB] TO DISK = 'D:\Backups\ProdDB_Full.bak'
  WITH COMPRESSION, CHECKSUM;

-- Differential daily (Mon-Sat)
BACKUP DATABASE [ProdDB] TO DISK = 'D:\Backups\ProdDB_Diff.bak'
  WITH DIFFERENTIAL, COMPRESSION;

-- Transaction log every 15 minutes (for RPO <15 min)
BACKUP LOG [ProdDB] TO DISK = 'D:\Backups\ProdDB_Log.bak';
Use SQL Server Agent Jobs to automate, and Ola Hallengren's maintenance scripts (industry standard) for intelligent backup management with automatic cleanup.

Write a Bash script that: checks disk usage across servers, sends an alert if >80%, and auto-cleans old logs.
HardBashAutomation
Optimum Answer
#!/bin/bash
# disk_monitor.sh — Run via cron: */15 * * * * /opt/scripts/disk_monitor.sh

THRESHOLD=80
ALERT_EMAIL="devops@company.com"
LOG_DIR="/var/log/app"
LOG_RETENTION_DAYS=30

check_disk() {
    local mount=$1
    local usage=$(df -h "$mount" | awk 'NR==2 {gsub(/%/,""); print $5}')

    if [ "$usage" -ge "$THRESHOLD" ]; then
        local hostname=$(hostname -f)
        local msg="⚠️ DISK ALERT: ${hostname}:${mount} at ${usage}% (threshold: ${THRESHOLD}%)"

        # Send email alert
        echo "$msg" | mail -s "[DISK ALERT] ${hostname}" "$ALERT_EMAIL"

        # Also send to Slack
        curl -s -X POST "$SLACK_WEBHOOK" \
          -H 'Content-type: application/json' \
          --data "{\"text\": \"${msg}\"}"

        echo "$(date '+%Y-%m-%d %H:%M:%S') $msg" >> /var/log/disk_alerts.log

        # Auto-clean old log files to recover space
        echo "Cleaning logs older than ${LOG_RETENTION_DAYS} days in ${LOG_DIR}..."
        find "$LOG_DIR" -name "*.log" -mtime +${LOG_RETENTION_DAYS} -delete
        find "$LOG_DIR" -name "*.log.gz" -mtime +${LOG_RETENTION_DAYS} -delete

        # Also clean rotated systemd journal logs
        journalctl --vacuum-time=30d 2>/dev/null

        local new_usage=$(df -h "$mount" | awk 'NR==2 {gsub(/%/,""); print $5}')
        echo "Disk usage after cleanup: ${new_usage}%"
    fi
}

# Check all mounted filesystems
df -h | awk 'NR>1 && $6 != "tmpfs" {print $6}' | while read mount; do
    check_disk "$mount"
done

import boto3
from datetime import datetime, timedelta

ec2 = boto3.client('ec2', region_name='ap-south-1')
ce  = boto3.client('ce',  region_name='us-east-1')  # Cost Explorer always us-east-1

def list_instances():
    resp = ec2.describe_instances()
    instances = []
    for r in resp['Reservations']:
        for i in r['Instances']:
            tags = {t['Key']: t['Value'] for t in i.get('Tags', [])}
            instances.append({
                'id':    i['InstanceId'],
                'type':  i['InstanceType'],
                'state': i['State']['Name'],
                'name':  tags.get('Name', 'N/A'),
                'env':   tags.get('env', 'N/A'),
            })
    return instances

def stop_dev_instances():
    resp = ec2.describe_instances(
        Filters=[
            {'Name': 'tag:env', 'Values': ['dev']},
            {'Name': 'instance-state-name', 'Values': ['running']}
        ]
    )
    ids = [i['InstanceId']
           for r in resp['Reservations']
           for i in r['Instances']]
    if ids:
        ec2.stop_instances(InstanceIds=ids)
        print(f"Stopped {len(ids)} dev instances: {ids}")

def cost_report(days=30):
    end   = datetime.today().strftime('%Y-%m-%d')
    start = (datetime.today() - timedelta(days=days)).strftime('%Y-%m-%d')
    resp = ce.get_cost_and_usage(
        TimePeriod={'Start': start, 'End': end},
        Granularity='MONTHLY',
        Metrics=['UnblendedCost'],
        GroupBy=[{'Type': 'DIMENSION', 'Key': 'SERVICE'}]
    )
    for result in resp['ResultsByTime']:
        for group in result['Groups']:
            svc  = group['Keys'][0]
            cost = group['Metrics']['UnblendedCost']['Amount']
            print(f"{svc}: ${float(cost):.2f}")

if __name__ == '__main__':
    print("=== EC2 Instances ===")
    for i in list_instances(): print(i)
    stop_dev_instances()
    print("\n=== 30-Day Cost Report ===")
    cost_report()
Q36
What is a FortiGate firewall? Explain firewall policies, VPN configuration (IPSec/SSL), and FortiGate in a datacenter context.
MediumFortiGateOn-PremiseSecurity
Optimum Answer
FortiGate is Fortinet's Next-Generation Firewall (NGFW). Used for perimeter security in on-premise datacenters. Combines traditional firewall, IPS (Intrusion Prevention), web filtering, SSL inspection, and SD-WAN in one device.

Firewall Policy structure:
  • Policies = rules evaluated top-to-bottom (first match wins, similar to AWS SGs but stateful and explicit deny)
  • Each policy: Source (interface + address) → Destination (interface + address) → Service (port/protocol) → Action (Accept/Deny) + Security profiles (IPS, AV, Web Filter)
  • NAT policies for outbound internet (source NAT) and inbound services (destination NAT / VIP)
VPN types on FortiGate:
  • IPSec Site-to-Site VPN: Connects two offices/datacenters. Phase 1 (IKE — key exchange, authentication) + Phase 2 (IPSec tunnel parameters). Used for branch office to HQ connectivity or on-prem to AWS/GCP (via AWS VPN Gateway).
  • SSL VPN: Remote user access. Client installs FortiClient VPN. User authenticates → gets VPN IP → tunneled access to internal resources. Supports 2FA (FortiToken).
HA setup: Two FortiGates in Active-Passive cluster. Heartbeat between units. If primary fails, secondary takes over in ~2-3 seconds with stateful session table sync.

Q37
How do you design a backup and disaster recovery strategy for on-premise infrastructure including NAS, rack servers, and VMs?
HardBackupDROn-Prem
Optimum Answer
3-2-1 Backup Rule: 3 copies of data, 2 different media types, 1 offsite copy. This is the baseline.

On-premise backup strategy:
  • VMs (Hyper-V / VMware): Use Veeam Backup & Replication — agent-less backup of entire VM snapshots. Schedule: Full backup weekly, incremental daily. Store on NAS (primary) + replicate to offsite NAS or cloud (secondary).
  • NAS (Network Attached Storage): RAID 6 (survives 2 disk failures) for the NAS array itself. NAS-to-NAS replication to offsite location. Periodic offsite tape rotation for long-term archival.
  • Database servers: Native DB backups (SQL Server AG + transaction logs, PostgreSQL pg_dump or WAL archiving) → NAS → S3 Glacier.
  • Rack servers (bare metal): Image backups using Clonezilla or Acronis. Store on NAS. For recovery: PXE boot from backup image.
DR testing: Recovery is only as good as the last test. Quarterly DR drills — restore a VM from backup, verify database integrity, test RTO against SLA. Document the playbook.

Cloud offsite with AWS: AWS Storage Gateway → Tape Gateway mode acts as a virtual tape library, writing directly to S3 Glacier. Or use AWS Backup for hybrid on-prem + cloud backup management.

PM2 vs systemd:
  • PM2: Node.js specific, cluster mode, built-in log management, hot reload. Great for Node.js apps.
  • systemd: OS-level service manager. Language-agnostic. More reliable for production (systemd is battle-tested, PM2 is an extra abstraction layer). For critical production services, some teams prefer systemd + a single Node process (no cluster) and put Nginx in front for load distribution.
Best practice: Use pm2 startup systemd so PM2 itself is managed by systemd — you get the best of both worlds.

Design a high-availability architecture combining on-premise infrastructure with AWS cloud (hybrid cloud). Include network connectivity, failover, and monitoring.
HardHybrid CloudArchitecture
Optimum Answer
Hybrid architecture design:

Network connectivity (on-prem ↔ AWS):
  • AWS Direct Connect: Dedicated 1–100Gbps private connection from on-prem datacenter to AWS. No public internet — lower latency, consistent bandwidth. Best for production workloads with high data transfer.
  • AWS Site-to-Site VPN: IPSec VPN over internet as backup path or for lower-bandwidth needs. Used as Direct Connect failover.
  • AWS Transit Gateway: Hub connecting multiple VPCs + on-prem via single connection (instead of individual VPC peering).
Failover strategy:
  • Active workloads on on-prem servers; AWS is warm standby (Pilot Light)
  • Route 53 health checks monitor on-prem endpoints; on failure, DNS automatically routes to AWS (failover routing policy)
  • Data replication: AWS Storage Gateway, AWS DataSync, or database replication (SQL Server log shipping to RDS, PostgreSQL streaming replication)
Identity integration:
  • AWS Directory Service — AD Connector links AWS IAM to on-prem Active Directory. Users log in to AWS Console with their AD credentials.
Monitoring — unified view:
  • CloudWatch Agent on on-prem servers → sends metrics/logs to CloudWatch
  • Or: Prometheus + Grafana (on-prem) scraping both on-prem exporters and AWS CloudWatch exporter in one dashboard
Your experience: Map your 3-tier app VPC design + blue-green migration to this architecture. That work was the cloud side of exactly this pattern.
