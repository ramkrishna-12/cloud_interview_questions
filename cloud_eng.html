Q1
Explain the difference between Security Groups and NACLs in AWS VPC. When would you use each?
MediumVPCNetworking
Optimum Answer
Security Groups are stateful, instance-level firewalls. If you allow inbound traffic on port 80, the return traffic is automatically allowed — you don't need to write an explicit outbound rule. They only support allow rules.

NACLs (Network Access Control Lists) are stateless, subnet-level firewalls. You must explicitly allow both inbound AND outbound rules. They support both allow and deny rules, and rules are evaluated in numeric order (lowest first).

When to use each:
  • Use Security Groups for fine-grained, per-instance control — e.g., allow only your ALB to talk to your EC2 on port 8000.
  • Use NACLs for broad subnet-level blocking — e.g., block a known malicious IP range at the subnet boundary, or add a defense-in-depth layer.
Ravan's context: In your 3-tier compliance app, you likely used SGs to allow ALB → EC2 on app port, and EC2 → RDS on 3306. Mention that pattern if asked for a real example.
Q2
What is the difference between an ALB and an NLB? When would you pick NLB over ALB?
MediumLoad Balancing
Optimum Answer
ALB (Application Load Balancer) operates at Layer 7 (HTTP/HTTPS). It understands HTTP headers, paths, host-based routing, and can route /api/* to one target group and /static/* to another. It supports WebSockets and gRPC. Best for microservices and HTTP workloads.

NLB (Network Load Balancer) operates at Layer 4 (TCP/UDP/TLS). It's ultra-low latency, preserves the client's source IP, and can handle millions of requests per second. Best for: real-time gaming, IoT, financial trading, or any TCP workload where HTTP-level features aren't needed.

Pick NLB when: you need static IP addresses (NLB gives you one per AZ), sub-millisecond latency, TLS passthrough to backend, or are dealing with non-HTTP protocols.
Q3
Describe how AWS IAM Roles differ from IAM Users, and why roles are preferred for EC2 instances and Lambda functions.
EasyIAMSecurity
Optimum Answer
IAM Users have permanent long-term credentials (access key + secret). They represent humans or external systems. Storing these on EC2 is a major security risk — if the instance is compromised, the keys are exposed.

IAM Roles have no permanent credentials. Instead, they issue temporary security tokens (via AWS STS) that auto-rotate. When you attach a role to an EC2 instance (Instance Profile), the application can call boto3 or the AWS SDK without hardcoding any credentials — it fetches tokens from the 169.254.169.254 metadata endpoint automatically.

Why roles are preferred:
  • No credential management overhead
  • Tokens auto-expire — reduces blast radius of a breach
  • Follows the principle of least privilege at the service level
  • Same pattern works across EC2, Lambda, ECS tasks, and EKS pods
Q4
You need to deploy a FastAPI application on EC2 with zero-downtime deployments. Walk through your architecture.
HardEC2DeploymentALB
Optimum Answer
Architecture:
  • Place EC2 instances in an Auto Scaling Group (ASG) across 2+ AZs
  • Put an ALB in front with a Target Group pointing to the ASG
  • Use Launch Templates (not Launch Configurations) — easier to version
Zero-downtime deployment flow:
  1. Create a new Launch Template version with the updated AMI or user-data
  2. Trigger an Instance Refresh on the ASG with a minimum healthy percentage of 50–100%
  3. ASG terminates old instances only after new ones pass ALB health checks
  4. ALB connection draining (default 300s) ensures in-flight requests complete
Alternative — Blue/Green: Spin up a new ASG (Green), register it with a second Target Group, then switch ALB listener rules from Blue TG to Green TG in one atomic operation. This is the pattern I used in my blue-green server migration with Route 53 weighted routing as the cutover mechanism.

Q5
What is S3 Versioning, and how would you use S3 Lifecycle Policies to manage storage costs?
EasyS3Cost Optimization
Optimum Answer
S3 Versioning keeps every version of an object, including all overwrites and deletes (which create a delete marker). Useful for compliance, accidental deletion recovery, and audit trails — relevant for a textile fintech platform storing proposals and pricing data.

Lifecycle Policies automate cost optimization:
  • Transition rules: Move objects to cheaper storage tiers over time — e.g., Standard → Standard-IA after 30 days → Glacier Instant Retrieval after 90 days → Glacier Deep Archive after 365 days
  • Expiration rules: Auto-delete old object versions after N days to prevent version bloat
  • Incomplete multipart uploads: Auto-abort uploads older than 7 days (often overlooked cost leaker)
For the textile platform, I'd version pricing proposals in S3, transition them to Glacier after 180 days, and delete versions older than 3 years to meet compliance archival requirements at minimal cost.
Q6
Explain RDS Multi-AZ vs Read Replicas. How do they serve different purposes?
MediumRDSDatabase
Optimum Answer
Multi-AZ is for High Availability / Disaster Recovery. AWS maintains a synchronous standby replica in a different AZ. If the primary fails, automatic failover happens in ~60–120 seconds with no data loss (RPO=0). The standby is NOT readable — it exists solely for failover.

Read Replicas are for Read Scalability / Performance. They use asynchronous replication (small replication lag). You can have up to 15 read replicas for PostgreSQL. Your app explicitly sends SELECT queries to the replica endpoint. They can be in the same region, cross-region, or even promoted to a standalone DB.

Key difference:
  • Multi-AZ = passive standby, automatic failover, synchronous replication
  • Read Replica = active readable copy, app-driven, asynchronous, used for offloading analytics or reporting queries
For the textile platform's PostgreSQL, I'd use Multi-AZ for the main transactional DB, and a read replica for the analytics/ESG dashboard queries to avoid impacting OLTP performance.
Q7
How does AWS Auto Scaling work? Explain scaling policies: Target Tracking vs Step Scaling vs Scheduled.
MediumAuto ScalingEC2
Optimum Answer
Auto Scaling Groups maintain a fleet of EC2 instances between a min/desired/max count, using health checks (EC2 + ALB) to replace unhealthy instances automatically.

Target Tracking — simplest. You say "keep average CPU at 60%". AWS automatically adjusts the fleet to maintain that metric. Equivalent to a thermostat. Best for most production workloads.

Step Scaling — more granular. Define steps: "if CPU > 70%, add 2 instances; if CPU > 90%, add 4 instances". More responsive to sudden spikes but more complex to configure.

Scheduled Scaling — proactive. "Every weekday at 8 AM, set min=10 instances; at 8 PM, set min=2". Perfect for predictable load patterns like business-hours traffic.

Your project context: In your Terraform + Auto Scaling project, you likely used Target Tracking — mention your experience with Launch Templates and ASG via Terraform if relevant.
Q8
What is AWS Secrets Manager vs SSM Parameter Store? Which would you use for storing database credentials in a FastAPI app?
MediumSecuritySecrets
Optimum Answer
SSM Parameter Store — free for standard parameters, supports String, StringList, and SecureString (encrypted with KMS). Good for config values, feature flags, non-rotating secrets.

Secrets Manager — paid (~$0.40/secret/month), but adds automatic secret rotation natively integrated with RDS. It also handles the rotation Lambda for you. Best for database passwords, API keys that rotate.

For a FastAPI + PostgreSQL app: I'd use Secrets Manager for DB credentials because of auto-rotation. In FastAPI: 
import boto3, json

def get_db_url():
    client = boto3.client('secretsmanager', region_name='ap-south-1')
    secret = client.get_secret_value(SecretId='prod/textile-db')
    creds = json.loads(secret['SecretString'])
    return f"postgresql://{creds['username']}:{creds['password']}@{creds['host']}/textile"
Fetch credentials at app startup (or cache with a TTL) — never hardcode in environment variables.

Q9
Compare GCP Compute Engine, GKE, Cloud Run, and App Engine. When would you choose each?
MediumGCP Compute
Optimum Answer
  • Compute Engine (IaaS) — Raw VMs. Full control over OS, networking, software. Use when you need custom OS configs, Windows workloads, or lift-and-shift migrations. Highest management overhead.
  • GKE (Google Kubernetes Engine) — Managed Kubernetes. Use for containerized microservices at scale, complex orchestration needs, multi-container workloads. Middle ground: you manage workloads, GCP manages the control plane.
  • Cloud Run (Serverless containers) — Runs any container, auto-scales to zero, billed per request. Perfect for stateless HTTP workloads like a FastAPI service. No cluster management. Best cost-efficiency for variable traffic.
  • App Engine — PaaS, fully managed. Opinionated platform for web apps. Standard env = zero cold start, sandbox; Flexible env = custom runtimes. Use for simple web apps where you want zero infra management.
For WebSpider's FastAPI: Cloud Run is the simplest GCP deployment — containerize the FastAPI app, push to Artifact Registry, deploy to Cloud Run with --allow-unauthenticated or behind Cloud Endpoints.
Q10
What is GCP's IAM structure? Explain the difference between Roles, Service Accounts, and Policies.
EasyIAMSecurity
Optimum Answer
GCP IAM follows the model: Who (identity) can do What (role) on Which resource (policy binding).

  • Identities: Google accounts (human users), Google Groups, Service Accounts (non-human workload identities), Cloud Identity domains
  • Roles: Collections of permissions. Three types: Primitive (Owner/Editor/Viewer — avoid in prod), Predefined (roles/storage.objectCreator), Custom (create your own)
  • Policy Binding: Attaches a role to an identity at a resource level (organization → folder → project → resource). Policies are additive — no explicit deny (unlike AWS).
  • Service Accounts: Identity for GCE instances, Cloud Run, Cloud Functions to authenticate to other GCP services — equivalent to AWS IAM Roles for EC2.
GCP has no explicit deny rules — if a user has a permission at any level, it applies. Plan IAM at the project/folder level carefully.

Q12
What is BigQuery and how does it differ from Cloud SQL for analytical workloads?
EasyBigQueryAnalytics
Optimum Answer
Cloud SQL — Managed relational DB (PostgreSQL/MySQL). Optimized for OLTP (Online Transaction Processing): many small, fast reads/writes. Row-based storage. Scales vertically.

BigQuery — Serverless, columnar data warehouse. Optimized for OLAP (Online Analytical Processing): large-scale aggregations over billions of rows. No servers to manage — you just run SQL and pay per TB scanned. Scales massively in parallel.

For the textile platform's ESG tracking and order book analytics: Cloud SQL handles live transactional data. BigQuery handles historical trend analysis — e.g., "what's our average carbon footprint per yarn type over the last 3 years across all orders?" — a query that would kill a Cloud SQL instance but runs in seconds on BigQuery.
Q13
What is Google Cloud Pub/Sub and when would you use it in a data pipeline?
MediumPub/SubData Pipeline
Optimum Answer
Cloud Pub/Sub is a fully managed, asynchronous messaging service (like AWS SQS + SNS). Publishers send messages to a Topic; Subscribers pull from Subscriptions on that topic. Decouples services, handles backpressure, guarantees at-least-once delivery.

Use cases for the textile platform:
  • SAP ERP exports arrive → publish to Pub/Sub topic → Dataflow job consumes → loads into BigQuery and Cloud SQL
  • WGSN trend data webhook → Pub/Sub → Cloud Function → updates material recommendations
  • Celery can be replaced by Pub/Sub + Cloud Tasks for GCP-native async task queuing
Pub/Sub is useful when you need to fan out to multiple consumers (e.g., both the pricing engine and ESG tracker react to the same inventory update event).
Q14
How does GCP's VPC differ from AWS VPC? What is Shared VPC?
MediumVPCNetworking
Optimum Answer
Key differences:
  • GCP VPCs are global by default — subnets are regional, but a single VPC spans all regions. In AWS, a VPC is region-scoped.
  • GCP uses Firewall Rules (tag-based or service-account-based) instead of Security Groups + NACLs. Rules are applied at the network level, not the instance level.
  • GCP has no concept of internet gateways — internet access is controlled via route tables and Cloud NAT.
Shared VPC — A Host Project owns the VPC; Service Projects share it. This allows centralized network management with IAM-controlled access. Teams deploy resources (VMs, containers) into the shared VPC without managing their own network. Common in enterprises where networking is centralized (Network team manages Shared VPC, dev teams manage their service projects).

Write an optimized, production-grade Dockerfile for a FastAPI application.
HardDockerfileBest Practices
Optimum Answer
# ── Stage 1: Builder ──────────────────────────────────────
FROM python:3.11-slim AS builder

WORKDIR /app
COPY requirements.txt .

# Install deps into a prefix dir (avoids polluting system)
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ── Stage 2: Runtime ──────────────────────────────────────
FROM python:3.11-slim

# Non-root user for security
RUN groupadd -r appuser && useradd -r -g appuser appuser

WORKDIR /app

# Copy only installed packages from builder
COPY --from=builder /install /usr/local
COPY --chown=appuser:appuser . .

USER appuser

EXPOSE 8000

# Gunicorn with Uvicorn workers for production
CMD ["gunicorn", "main:app", "-k", "uvicorn.workers.UvicornWorker",
     "-w", "4", "-b", "0.0.0.0:8000", "--timeout", "120"]
Why this is optimized:
  • Multi-stage build — final image has no build tools (smaller, fewer vulnerabilities)
  • python:3.11-slim — minimal base image (~50MB vs 900MB for full Python)
  • Non-root user — follows security best practices
  • Gunicorn + UvicornWorker — production-grade async serving with process management
  • --no-cache-dir — avoids storing pip cache in the layer
Q16
What is Docker layer caching and how do you optimize Dockerfile layer order to maximize cache hits?
MediumLayer CachingPerformance
Optimum Answer
Docker caches each instruction as a layer. If an instruction and everything before it is unchanged, Docker reuses the cached layer — drastically speeding up builds. A cache miss on any layer invalidates all subsequent layers.

Key rule: Put things that change least frequently first, things that change most frequently last.

❌ Bad order: 
COPY . .                         # Code changes every commit → invalidates cache
RUN pip install -r requirements.txt  # Re-runs on EVERY code change!
✅ Good order: 
COPY requirements.txt .           # Only changes when deps change
RUN pip install -r requirements.txt  # Cached unless requirements.txt changes
COPY . .                         # App code — changes frequently, but pip already cached
This simple reordering can reduce build times from 3 minutes to 10 seconds on a typical Python app.

Q18
Explain Docker networking modes: bridge, host, overlay, and none. When would you use each?
MediumNetworking
Optimum Answer
  • Bridge (default) — Creates a private internal network on the host. Containers on the same bridge network communicate by container name (Docker DNS). Use for single-host multi-container apps (docker-compose default). Port mapping required for external access.
  • Host — Container shares the host's network namespace directly. No network isolation, no port mapping overhead. Use when performance is critical (e.g., high-throughput network benchmarking) or for privileged monitoring tools. Security risk — don't use for app containers.
  • Overlay — Multi-host networking for Docker Swarm / Kubernetes. Spans multiple Docker hosts. Containers on different hosts communicate as if on the same LAN. Use in production clusters.
  • None — Container has no network access. Use for batch jobs that don't need network, or for maximum isolation during security-sensitive processing (e.g., sandboxed code execution).
Q19
What is a Docker volume vs a bind mount? How do you handle persistent data for a PostgreSQL container in production?
EasyVolumesStorage
Optimum Answer
Named Volumes — Managed by Docker, stored in Docker's storage area (/var/lib/docker/volumes/). Portable, easy to backup, not tied to host path structure. Preferred for production databases.

Bind Mounts — Maps a specific host directory into the container. Host-dependent, fragile in production. Good for development (mount your source code for hot-reload) but bad for production databases.

For PostgreSQL in production: 
volumes:
  pgdata:
    driver: local  # Or use a cloud volume driver like EBS CSI

db:
  image: postgres:15
  volumes:
    - pgdata:/var/lib/postgresql/data  # Named volume
In production on AWS, run RDS instead of PostgreSQL in Docker — managed backups, Multi-AZ, and automatic minor version patching without Docker volume management headaches.
Q20
How would you debug a container that is crashing on startup (CrashLoopBackOff / Exit Code 1)?
MediumDebuggingTroubleshooting
Optimum Answer
Systematic debugging approach: 
# 1. Check the logs before it dies
docker logs <container_id> --tail=50

# 2. Override entrypoint to keep it alive, then inspect
docker run -it --entrypoint /bin/bash my-image

# 3. Check exit code
docker inspect <container_id> --format='{{.State.ExitCode}}'

# 4. Run with --rm and debug mode
docker run --rm -e DEBUG=true my-image

# 5. Check resource limits (OOM kill = exit code 137)
docker stats <container_id>
Common causes:
  • Missing environment variables (DB_URL not set)
  • Port already in use on the host
  • Dependency not ready (database not up yet — use healthchecks + depends_on)
  • Wrong CMD / entrypoint path in Dockerfile
  • OOM kill (exit 137) — increase memory limit or fix memory leak

Q23
What is the difference between Blue/Green, Canary, and Rolling deployments? When would you use each?
HardDeployment Strategy
Optimum Answer
  • Rolling: Gradually replace old instances with new ones. Zero downtime, but both versions run simultaneously during the rollout. Rollback is slow (roll back forward). Best for stateless apps with backward-compatible changes.
  • Blue/Green: Full parallel environment. Run both Blue (old) and Green (new) simultaneously. Switch traffic all at once (or via weighted routing). Instant rollback — just switch back. Requires 2x infrastructure cost during transition. Best for major releases, database schema changes, or when you need instant rollback.
  • Canary: Route a small percentage (5–10%) of traffic to the new version. Monitor error rates and latency. Gradually increase traffic if healthy. Limits blast radius of bugs. Best for high-traffic production systems where you want data-driven release confidence.
Your experience: You executed Blue/Green migration with Route 53 weighted routing. Mention that — it's textbook canary/blue-green hybrid and directly applicable to this role.
Q24
How do you manage secrets securely in a GitHub Actions pipeline? What are the common mistakes?
MediumSecuritySecrets
Optimum Answer
Best practices:
  • Store secrets in GitHub Secrets (Settings → Secrets and variables → Actions) — they are encrypted at rest and masked in logs
  • Use OIDC (OpenID Connect) for AWS instead of storing access keys — GitHub Actions can assume an AWS IAM Role via federated identity, zero static credentials
  • Use environment-scoped secrets (prod vs staging) with required reviewers on the prod environment
  • Rotate secrets regularly; audit secret access with GitHub's secret scanning alerts

  • Storing secrets in environment variables at the job level unnecessarily
  • Using long-lived AWS access keys instead of OIDC roles
  • Checking in .env files or terraform.tfvars with real values

Q26
What is GitLab CI/CD and how does it compare to GitHub Actions? Describe the .gitlab-ci.yml structure.
MediumGitLab CI

Q27
What is Terraform state and how do you manage it in a team? What is state locking?
MediumTerraformState Management
Optimum Answer
Terraform state (terraform.tfstate) is the source of truth about what infrastructure Terraform has created. It maps your config to real cloud resources. Without state, Terraform can't determine what needs to be changed vs created.

Remote state for teams: Store state in an S3 bucket (never commit it to git — it contains sensitive values): 
terraform {
  backend "s3" {
    bucket         = "textile-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "ap-south-1"
    encrypt        = true
    dynamodb_table = "textile-tf-lock"
  }
}
State locking — When multiple engineers run terraform apply simultaneously, they'd corrupt the state file. DynamoDB is used as a lock table. Terraform writes a lock item before modifying state and deletes it after. If another apply is running, it sees the lock and waits or fails gracefully.

Q29
Explain Active Directory Domain Services (AD DS). What is the difference between a Domain, Tree, and Forest?
MediumActive Directory
Optimum Answer
AD DS is Microsoft's directory service that provides centralized authentication (Kerberos/NTLM), authorization, and resource management for Windows environments.

  • Domain — The basic administrative unit. A collection of objects (users, computers, groups, policies) sharing a common database, security policies, and namespace (e.g., textile.local). Each domain has at least one Domain Controller (DC).
  • Tree — Multiple domains sharing a contiguous DNS namespace and connected by transitive two-way trusts. E.g., textile.local → sales.textile.local → india.sales.textile.local
  • Forest — The top-level container. A collection of one or more trees sharing a common schema, configuration, and Global Catalog, but NOT necessarily a common namespace. The first domain in a forest is the Forest Root. All domains within a forest implicitly trust each other.
Group Policy (GPO) is layered on top: LSDOU order (Local → Site → Domain → OU), with later policies winning.

Q31
What is WSUS and how do you manage Windows Update in an enterprise environment?
EasyWSUSPatch Management
Optimum Answer
WSUS (Windows Server Update Services) is a free server role that downloads updates from Microsoft and distributes them to clients on your network. Centralizes control over which updates are approved and deployed, reducing internet bandwidth and giving IT control over patch timing.

Enterprise WSUS workflow:
  • WSUS server syncs with Microsoft Update (scheduled, e.g., nightly)
  • Admins review and approve updates per computer group (e.g., Dev → Test → Production — staggered by weeks)
  • GPO points client machines to the WSUS server (Computer Config → Admin Templates → Windows Components → Windows Update)
  • Clients report compliance; WSUS generates patch status reports
Modern alternative: Microsoft Intune + Windows Update for Business (WUfB) for cloud-managed endpoints. WSUS is still prevalent in on-prem enterprise environments.

Q34
How do you implement JWT authentication with refresh tokens in FastAPI? Describe the security model.
HardFastAPIJWTAuth
Optimum Answer
from fastapi.security import OAuth2PasswordBearer
from jose import JWTError, jwt
from datetime import datetime, timedelta

SECRET_KEY = "from-secrets-manager"
ACCESS_TOKEN_EXPIRE = timedelta(minutes=30)
REFRESH_TOKEN_EXPIRE = timedelta(days=7)

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

def create_token(data: dict, expires: timedelta) -> str:
    payload = data.copy()
    payload["exp"] = datetime.utcnow() + expires
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

@app.post("/auth/token")
async def login(form: OAuth2PasswordRequestForm = Depends()):
    user = authenticate_user(form.username, form.password)
    if not user: raise HTTPException(401)
    return {
        "access_token": create_token({"sub": user.id, "role": user.role},
                                      ACCESS_TOKEN_EXPIRE),
        "refresh_token": create_token({"sub": user.id, "type": "refresh"},
                                       REFRESH_TOKEN_EXPIRE),
    }
Security model: Access tokens are short-lived (30 min), stored in memory (not localStorage). Refresh tokens are longer-lived (7 days), stored in httpOnly cookies to prevent XSS. Refresh tokens are rotated on each use and stored in Redis for revocation capability. RBAC is embedded in the JWT claims (role: "pricing_manager") and enforced per endpoint.
